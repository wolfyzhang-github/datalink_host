## 3. 已确认的需求基线

### 3.1 总体定位

本项目是一个面向高精度长基线光纤应变传感系统的上位机软件，核心职责包括：

- 接收设备端实时数据
- 对 8 通道数据进行相位展开与降采样
- 实时显示波形与 PSD
- 本地保存 MiniSEED 文件
- 向远端 ringserver 以 DataLink 协议推送数据
- 提供独立控制链路
- 提供 GUI 与配套调试工具

### 3.2 协议与存储基线

- 数据接入协议：`TCP`
- 数据接入格式：以 `python-需求文档.docx` 的 TCP 段落为准
- 远程控制：单独一条控制连接，不与数据连接复用
- 远程传输协议：`DataLink`
- 远端服务端：标准 `ringserver`
- 本地存储格式：`MiniSEED`

### 3.3 GUI 基线

- 需要提供 Windows GUI 客户端
- 需要尽量跨平台，至少支持在 macOS 查看和验证界面与主要功能
- 界面不只是展示层，应承担连接管理、参数配置、状态监控、日志查看和调试入口

### 3.4 调试工具基线

项目需要配套以下调试能力：

- 运行日志
- 模拟发送端
- 模拟接收端
- 协议级联调工具
- 数据录制与回放能力

## 5. 关键约束与设计原则

### 5.1 实时性优先

系统首要目标是稳定连续运行，实时链路不能因为 GUI 刷新、文件写盘或远程发送而阻塞。

### 5.2 处理链路与界面解耦

采集、处理、存储、远传必须可在无 GUI 模式下独立运行。  
GUI 应作为控制与观测层，而不是系统主线程的宿主。

### 5.3 协议边界清晰

数据连接、控制连接、DataLink 推送、MiniSEED 落盘必须分别封装，避免后期联调时互相牵连。

### 5.4 先可联调，再优化

第一阶段优先打通：

- TCP 收数
- 数据解包
- 8 通道处理
- MiniSEED 落盘
- DataLink 推送
- GUI 观测

在链路稳定后，再做性能优化、体验优化和高级诊断功能。

## 7. 总体架构

建议采用“核心服务层 + 适配层 + GUI 层 + 调试工具层”的架构。

### 7.1 架构分层

1. 设备接入层
   - 数据 TCP 客户端
   - 控制 TCP 客户端或服务端适配器

2. 核心处理层
   - 解包与校验
   - 通道重组
   - 相位展开
   - 两路降采样
   - 队列与背压管理

3. 输出层
   - MiniSEED 写盘
   - DataLink 推送到 ringserver
   - GUI 可视化数据发布

4. 运维与调试层
   - 结构化日志
   - 指标统计
   - 模拟器
   - 录制与回放

5. 表现层
   - 桌面 GUI

### 7.2 推荐运行模型

- 网络接收、信号处理、文件写入、远程发送各自使用独立工作线程或任务
- 模块之间通过有界队列通信
- GUI 只订阅摘要状态和抽样后的显示数据
- 所有耗时操作禁止直接运行在 GUI 主线程

### 7.3 建议的数据流

`TCP 数据接收 -> 二进制解包 -> 8 通道重组 -> 相位展开 -> Data1/Data2 降采样 ->`

`GUI 显示发布`

`同时 -> MiniSEED 写盘`

`同时 -> DataLink 推送`

`同时 -> 状态统计与日志记录`

## 8. 数据连接协议设计

### 8.1 已确认的数据包字段

按 `python-需求文档.docx`，数据包字段结构为：

- 帧头：`U32`，4 字节
- 采样率：`Double`，8 字节
- 传输字节数：`I32`，4 字节
- 数据：`Double[]`

### 8.2 当前必须补充确认的细节

原始文档只确定了字段类型，没有完全覆盖以下关键细节。后续联调前必须形成协议补充说明：

- 字节序：大端、小端，或 LabVIEW 默认打包方式
- 帧头常量值
- `传输字节数` 的定义是否只覆盖 `数据` 段，还是覆盖整个 payload
- 单包是否固定为“1 秒数据”
- 8 通道数据在 `Double[]` 中的排列方式
  - 按通道连续：`CH1 全部 -> CH2 全部 -> ...`
  - 按采样交织：`CH1, CH2, ..., CH8, 下一采样点`
- 断线重连后的首包对齐要求
- 是否存在空包、补零包或心跳包

### 8.3 当前实现建议

在正式联调前，接收端实现不应把以上未确认项写死，而应采用以下策略：

- 字节序支持配置
- 帧头常量支持配置
- 通道布局支持配置
- 当 `传输字节数` 与包长不一致时输出诊断日志
- 保留原始包十六进制抓包能力，便于与 LabVIEW 侧对齐

### 8.4 业务侧固定约束

即使 TCP 包格式来自第二份文档，业务上仍以主需求中的 8 通道系统为准，因此：

- 系统默认按 8 通道处理
- 通道数应允许通过配置覆盖
- 如果 TCP 包本身不携带通道数，则由本地配置或控制连接下发

## 9. 控制连接设计

### 9.1 基本原则

- 控制链路与数据链路分离
- 控制命令以可读、可调试为优先
- 控制命令必须支持应答
- 所有控制变更必须记录日志

### 9.2 推荐控制协议

建议采用基于 TCP 的行分隔 `JSON` 协议，即每条消息为一行 JSON。

理由：

- 人工调试方便
- Wireshark / `nc` / 自定义工具都容易联调
- 对未来新增控制项兼容性好

### 9.3 控制消息建议字段

- `type`：消息类型，如 `set_config`、`get_status`、`start`、`stop`
- `request_id`：请求唯一标识
- `timestamp`：发送时间
- `payload`：业务内容

### 9.4 首批控制能力建议

- 设置采样率相关参数
- 设置 Data1 / Data2 目标采样率
- 设置存储开关
- 设置存储路径
- 设置台站号、位置码、通道码映射
- 设置单文件存储时长
- 设置 DataLink 推送开关
- 查询系统状态
- 查询队列积压
- 查询最近错误

## 10. 核心处理链路设计

### 10.1 解包与重组

职责：

- TCP 粘包/半包处理
- 帧头校验
- 长度校验
- 将 `Double[]` 重组为多通道数组

输出建议格式：

- `numpy.ndarray`
- 形状建议统一为 `(channels, samples)`，减少后续处理分支

### 10.2 相位展开

职责：

- 对 8 通道分别进行相位展开

实现建议：

- 先以 `numpy.unwrap` 或等效算法实现第一版
- 保留独立策略接口，便于后续替换为甲方认可的实现

### 10.3 两路降采样

职责：

- 基于相位展开后的数据生成 `Data1` 和 `Data2`
- 两路目标采样率独立可调
- 采用平均法降采样

实现要求：

- 目标采样率支持 `1/10/100/200 Hz`，并保留扩展能力
- 不能跨包丢失窗口上下文
- 配置变更时要平滑切换，避免边界跳变过大

### 10.4 PSD 计算

职责：

- 对选定通道进行 PSD 估计

实现建议：

- 基于滑动时间窗
- GUI 可配置时间窗长度
- 计算任务与主处理链路隔离，避免影响实时性

## 11. 本地存储设计

### 11.1 存储对象

- `Data1` 写入一套 MiniSEED
- `Data2` 写入一套 MiniSEED

### 11.2 文件组织

沿用原需求思路：

- 不同数据组写入不同目录
- 每个通道独立文件
- 文件名支持 Network / Station / Location / Channel 可配置

### 11.3 存储边界

需支持：

- 存储根路径配置
- 单文件存储时长配置
- 启停控制
- 异常恢复
- 磁盘写入失败告警

### 11.4 时间基准

原需求提到 UTC 时间来自串口返回。  
当前文档尚未覆盖串口链路的正式设计，因此在实现上应预留时间源接口，支持：

- 外部 UTC 时间输入
- 本机系统时间兜底

若后续串口时间同步是硬要求，则单独补充一个“时间服务模块”。

## 12. DataLink / ringserver 远传设计

### 12.1 目标

将指定数据流推送到标准 `ringserver` 服务端，协议为 `DataLink`。

### 12.2 设计边界

本项目负责：

- DataLink 客户端连接管理
- 数据组织与发送
- 失败重连
- 发送状态统计
- 发送错误日志

本项目暂不负责：

- ringserver 部署与运维
- 服务端长期归档策略
- 服务端元数据管理

### 12.3 数据组织建议

远传部分建议发送规范化的 MiniSEED 记录，而不是发送自定义裸二进制。

理由：

- ringserver 对 MiniSEED 生态最成熟
- 本地存储与远传可共享数据组织逻辑
- 有利于后续联调、回放和兼容工具链

### 12.4 发送策略建议

- 优先发送 `Data1`
- `Data2` 是否远传做成配置项
- 按记录批次提交
- 失败后有限重试
- 超过阈值时丢弃最旧待发缓存，并告警

### 12.5 必须纳入监控的指标

- 当前连接状态
- 最近连接时间
- 最近发送成功时间
- 发送速率
- 累计发送包数/字节数
- 重连次数
- 待发送队列长度

## 17. 分阶段开发建议

### 第一阶段：链路打通

目标：

- TCP 收包
- 多通道重组
- 基础 GUI
- 日志
- 模拟发送端

交付标准：

- 能稳定接收模拟端连续发送的数据
- 能在 GUI 中看到 8 通道波形
- 能记录基础运行日志

### 第二阶段：处理与存储

目标：

- 相位展开
- 两路降采样
- MiniSEED 写盘
- 控制连接

交付标准：

- Data1 / Data2 可独立生成
- 存储参数可配置
- 输出文件结构符合约定

### 第三阶段：远传与诊断

目标：

- DataLink 对接 ringserver
- 发送状态监控
- 模拟接收端
- 录制与回放工具

交付标准：

- 能向 ringserver 稳定发送数据
- 断连后可自动恢复
- 能使用调试工具复现典型问题

### 第四阶段：联调与交付

目标：

- 与真实 LabVIEW / 设备联调
- 协议细节收敛
- 性能与稳定性修正
- 打包发布

交付标准：

- Windows 发行版可运行
- 关键功能具备操作文档
- 典型故障具备定位手段

## 18. 当前仍需补充确认的事项

以下事项虽然不阻碍搭建骨架，但会影响正式实现细节：

- TCP 数据包字节序
- 帧头固定值
- 8 通道数据在 payload 中的排列方式
- 控制连接中“谁主动连谁”的网络拓扑
- 串口 UTC 时间是否属于本期必做
- MiniSEED 记录长度、编码细节与命名最终规范
- DataLink 推送的数据流命名规则
- Data1 与 Data2 是否都需要远传

## 19. 当前开发建议结论

后续实现建议遵循以下路线：

- 先实现无 GUI 也可运行的核心服务
- 再接入 GUI
- 以模拟发送端驱动第一轮开发
- 把协议解析做成可配置
- 把 MiniSEED 作为本地存储与 DataLink 远传的统一中间格式
- 把日志、录包、回放作为第一阶段就纳入的基础设施，而不是最后补
